# Payment Domain — Aish Laundry App

**Step:** 1 — Product Requirement and Domain Model
**Status:** `NOT IMPLEMENTED` (documentation only)
**Canonical source:** [`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) v1.0.1
**Hard gate:** [DEC-0012](../decisions/DEC-0012-tenant-isolation-and-financial-integrity-hard-gate.md)
**State machines:**
[`../state-machines/PAYMENT_STATE_MACHINE.md`](../state-machines/PAYMENT_STATE_MACHINE.md),
[`../state-machines/REFUND_STATE_MACHINE.md`](../state-machines/REFUND_STATE_MACHINE.md)

Money that does not add up destroys trust faster than any missing feature. **Any financial integrity
failure is an automatic `NO-GO`** (`FIN-040`).

---

## 1. Scope

Payment and Receivables is the **sole writer of money** in the entire system. It owns `Payment`,
`Refund`, `Receivable`, and `CashierShift`, and it posts the entries generated by courier cash
collection and by loyalty and deposit movement.

Every other context — the unclaimed dashboard, the owner portfolio, every report — **reads** money
from here and never recomputes it (`FIN-023`, `UCL-014`, `UCL-024`).

---

## 2. Representation

> **Money is stored as integer Rupiah. Floating point is forbidden in every financial path.**
> (`FIN-001`, `FIN-002`)

- The smallest representable unit is Rp1. `Money` is a value object over an integer.
- No `float`, no `double`, no binary floating-point arithmetic anywhere in pricing, totals, discounts,
  taxes, payments, refunds, reconciliation, reporting, transport, or storage — **including any display
  path that round-trips through a float**.
- Rounding is **explicit**: a defined mode applied at a defined point, recorded in the price snapshot
  (`FIN-014`, `FIN-033`). It is never a language's default numeric behaviour and never happens
  implicitly at a display boundary.
- **Money is never inferred from a display string** (`FIN-034`). Rupiah formatting is a view concern
  applied to an integer; parsing a formatted string back into money is forbidden.
- Arithmetic is total and checked. Addition, subtraction, and multiplication by an integer quantity
  are permitted. Division is permitted **only** through an explicit allocation rule that distributes
  remainders deterministically so the parts always sum to the whole (`FIN-035`).

A float found anywhere in a money path is rejected outright. There is no "it works in practice"
exemption.

---

## 3. Idempotency

> **Payments are idempotent, keyed on a stable client-supplied `ClientReference`. The same logical
> payment submitted any number of times produces exactly one payment.** (`FIN-003`)

- The client generates the `ClientReference` **once**, before the first attempt, persists it with the
  queued operation, and **reuses it unchanged on every retry** (`OFF-001`). Regenerating it on retry
  defeats the entire mechanism and is rejected (`OFF-025`).
- Idempotency is a **server contract**, not a client trick: the server recognises a repeated
  reference and returns the **original result** rather than creating a second record (`OFF-017`).
- Suppression is **observable**. A suppressed duplicate emits `PaymentDuplicateSuppressed`; it does
  not silently do nothing, because a silently-vanishing payment attempt is indistinguishable from a
  lost one to the kasir standing at the counter.
- Concurrent operations on the same order or payment are serialized by database transaction or
  distributed lock, so a double tap cannot create a double payment (`FIN-016`).

**A duplicate payment produced by a retry is an automatic `NO-GO`** (`FIN-039`).

---

## 4. Gateway verification

- **Gateway callbacks are verified server-side**: signature, amount, currency, and status checked
  against the gateway, not trusted from the callback payload alone. Replays are rejected (`FIN-004`).
- **An order is never marked paid on a client claim** (`FIN-005`). A client may report; only the
  server may decide.
- An **offline device may record a payment intent**. It may **never** record a confirmed gateway
  payment (`FIN-019`). The product states honestly that gateway confirmation requires the network
  (`OFF-023`).
- A `PaymentIntentRecorded` is never displayed to a kasir as "paid". Sync state is visible at all
  times (`OFF-013`).
- A rejected callback is recorded and alerted. **It does not change the order's payment state.**

---

## 5. Corrections

> **Financial transactions are never deleted through ordinary UI. Corrections happen via reversal or
> adjustment entries that preserve the original record.** (`FIN-007`, `FIN-008`)

- There is **no** `DeletePayment` command, **no** `EditCapturedAmount` command, and **no**
  `AdjustVarianceToZero` command anywhere in the model. Their absence is the invariant.
- The ledger is append-only in effect: history is added to, never rewritten.
- **Refund and void require an explicit permission and a recorded reason**, with actor, timestamp,
  amount, and reason text (`FIN-006`). A refund is never silent (`FIN-021`).
- A refund never exceeds the captured amount of the payment it reverses, net of prior refunds
  (`FIN-020`). Concurrent refunds are serialized against the parent payment so two cannot jointly
  exceed it.
- Where tenant policy requires separation of duties, the approving actor may not be the requesting
  actor.

---

## 6. Cash handling

### 6.1 Cashier shift

- Shift closing **compares expected cash against actual cash and records the variance explicitly**
  (`FIN-024`).
- A variance beyond the tenant's configured threshold requires a recorded reason and an
  acknowledgement (`FIN-025`).
- A variance is **never** masked, auto-rounded away, absorbed, or suppressed from a report
  (`FIN-026`). A visible discrepancy is a feature; a hidden one is fraud-shaped.
- A closed shift is never reopened. A correction is an adjustment entry.

### 6.2 Courier cash

- Cash collected at the door is a financial transaction and inherits **every** rule in this document
  (`FIN-027`).
- It is tracked per courier, per shift or route, from collection to hand-over (`FIN-028`).
- Expected versus actual is compared explicitly; any variance is recorded and acknowledged, never
  written off silently (`FIN-029`).
- A settlement cannot be accepted while an unacknowledged variance exists (`DEL-030`).
- Collections are captured at the door — frequently offline — so each carries a `ClientReference` and
  is idempotent on it (`OFF-022`).

See [`PICKUP_DELIVERY_DOMAIN.md`](PICKUP_DELIVERY_DOMAIN.md) and
[`../state-machines/COURIER_SETTLEMENT_STATE_MACHINE.md`](../state-machines/COURIER_SETTLEMENT_STATE_MACHINE.md).

---

## 7. Receivables and held invoices

- A `Receivable` balance is derived **only** from posted financial entries (`FIN-022`).
- A **held invoice** is an unsettled invoice on laundry that remains uncollected. It is a headline
  figure on the unclaimed dashboard, and the dashboard **reads** it from here (`FIN-023`).
- A write-down is only ever an authorised, reasoned, audited adjustment entry. There is no silent
  write-off path.

---

## 8. Pricing history

> **Historical order prices are immune to price-list changes.** (`FIN-012`, `FIN-036`)

An order captures the price that applied when it was created. Editing the master price list changes
future orders only. Reprinting an old nota shows the old price. See
[`ORDER_DOMAIN.md`](ORDER_DOMAIN.md) §3.

---

## 9. Audit

- Every financial audit entry records **actor, tenant, outlet, timestamp, before and after amounts,
  and reason** (`FIN-031`).
- The audit write shares the transaction with the financial action. **If the audit entry cannot be
  written, the action does not proceed** (`FIN-038`). Fail closed.
- The financial audit trail is **append-only and not subject to log rotation** (`FIN-032`).
- `AuditActor` is never anonymous and never a bare "system" — a named process, a membership, or a
  platform admin under active audited impersonation.

---

## 10. Tenant rules

- Every financial record carries `TenantId` and outlet (`TEN-015`).
- **A financial query that is not tenant-scoped is simultaneously a security defect and a financial
  defect.** Both gates fire.
- Financial exports, report files, and reconciliation artefacts are tenant-scoped and carry the same
  access rules as the underlying records.
- Deposits and loyalty balances are **customer money** held by a specific tenant. A deposit with one
  tenant is never spendable with another (`FIN-037`).

---

## 11. Offline behaviour

- **The financial queue is never casually deleted.** It is not cleared by a cache wipe, a logout, a
  version upgrade, or a developer convenience button. Clearing a queued financial operation requires
  an explicit, permissioned, audited action (`OFF-004`, `OFF-024`).
- Financial operations are pruned from the queue **only after confirmed server acceptance**, never on
  a timer (`OFF-021`).
- **Payment conflicts are never silently overwritten.** Both values surface to a human with enough
  context to decide (`OFF-010`).
- **Conflicts affecting money escalate to a human, always.** A documented last-write rule is
  permissible only for non-financial metadata (`OFF-011`).
- Resolution records actor, timestamp, chosen value, and reason (`OFF-012`).

---

## 12. Testing expectation (later Steps)

Recorded now so that Step 5 cannot ship weaker. The Definition of Done for payment functionality
requires demonstrated tests for: retry idempotency producing exactly one payment; duplicate gateway
callback rejected; offline queue replay after reconnect producing no duplicate; refund permission
enforcement; a price-list change not affecting a historical order; and shift-closing variance
calculation. **Absence of these tests blocks the Step.**

No such test exists today.

---

## 13. NO-GO triggers

Any of the following stops work immediately, notifies the repository owner, and preserves evidence at
the exact commit SHA:

- a duplicate payment created by a retry (`FIN-039`);
- a float appearing anywhere in a money path (`FIN-002`);
- an order marked paid without server verification (`FIN-005`);
- a deletable financial transaction (`FIN-007`);
- a historical price mutating after a price-list edit (`FIN-012`);
- a courier or shift cash discrepancy the system cannot account for (`FIN-029`);
- a financial query that is not tenant-scoped (`TEN-030` and `FIN-040` jointly);
- a queued financial operation deleted without an audited, permissioned action (`OFF-024`).

---

The payment domain is `NOT IMPLEMENTED`. No payment, ledger, gateway integration, shift, or
