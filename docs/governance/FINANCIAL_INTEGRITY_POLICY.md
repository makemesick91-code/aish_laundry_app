# Financial Integrity Policy — Aish Laundry App

Canonical source: [`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) §16
Locked by [DEC-0012](../decisions/DEC-0012-tenant-isolation-and-financial-integrity-hard-gate.md)
Baseline date: 19 July 2026

**Any financial integrity failure is an automatic NO-GO.**

This is hard gate 2. It blocks merge, blocks release, blocks a GO tag, and is never waived, never
deferred, and never traded against a deadline.

---

## 1. Why this is a hard gate

A laundry business runs on thin margins and cash. If the software double-charges a customer, loses a
payment, silently overwrites a cash reconciliation, or lets an old nota change its price, the owner
cannot trust any number the product shows. Once financial trust is lost, no feature recovers it.

Money defects also compound: a duplicate payment created today is discovered at shift closing tonight, is
disputed by the customer tomorrow, and corrupts the monthly report forever. There is no version of
"we will reconcile it later" that is acceptable.

---

## 2. The canonical rules

### 2.1 Representation

1. **Money is stored as integer Rupiah.**
2. **Floating point is forbidden for financial transactions.** No `float`, no `double`, no
   binary-floating-point arithmetic anywhere in a money path — storage, computation, transport,
   serialisation, or a display helper that round-trips through a float.

### 2.2 Payment correctness

3. **Payments are idempotent.** The same logical payment submitted twice — by a retry, a queue replay, a
   double tap, a network timeout, or a duplicated callback — produces exactly one payment.
4. **Gateway callbacks are verified server-side**: signature verification, amount verification, and
   status verification against the gateway, never trust in the callback payload alone.
5. **An order is never marked paid on a client claim.** A client may report; only the server decides.

### 2.3 Corrections and audit

6. **Refund and void require an explicit permission and a recorded reason.**
7. **Financial transactions are never deleted through ordinary UI.**
8. **Corrections happen via reversal or adjustment entries**, preserving the original record and the full
   audit trail.

### 2.4 Pricing history

9. **Historical order prices are immune to price-list changes.** An order captures the price that applied
   when it was created; editing the price list affects future orders only.

### 2.5 Cash handling

10. **Shift closing compares expected cash against actual cash**, records the variance, and requires a
    reason beyond a configured threshold.
11. **Courier cash is reconciled** from collection to hand-over, with the courier accountable for the
    difference.

## 3. Implementation requirements

Binding from **Step 5 — POS, Order, and Payment Foundation**, with offline aspects from **Step 6** and
courier cash from **Step 8**.

### 3.1 Integer money

1. Money columns are integer types storing whole Rupiah. There are no minor units in Rupiah as used here;
   the integer is the amount.
2. Money crosses the API as an integer JSON number or an integer-valued string, never as a decimal
   fraction that a client might parse into a float.
3. Money is represented in application code by a dedicated value type that makes float construction
   impossible.
4. Percentage-based calculations — discounts, taxes, service charges — compute in integer arithmetic with
   an explicitly defined and documented rounding rule. Rounding is deterministic and identical on server
   and client.
5. Totals are computed from line items with an explicit rounding boundary, so that the sum of the parts
   always equals the recorded whole.
6. A lint or static check rejects floating-point types in money paths.

### 3.2 Idempotency

7. Every payment-creating operation carries a `client_reference` generated by the client before the
   request is sent ([`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) §13).
8. The server enforces uniqueness of `client_reference` within the tenant at the **database** level, so
   that a race cannot produce two rows.
9. A repeat request with the same `client_reference` returns the **original** result, and does not create
   a second payment.
10. Gateway callbacks are idempotent by gateway transaction identifier; a replayed callback changes
    nothing.
11. Idempotency applies equally to refunds, voids, and adjustments.

### 3.3 Gateway verification

12. Callback signatures are verified using the shared secret before any parsing of business meaning.
13. The amount in the callback is verified against the expected amount for the order.
14. Where the gateway supports it, the payment status is re-fetched from the gateway rather than being
    trusted from the callback.
15. A callback that fails any verification is recorded, alerted on, and never applied.
16. A payment intent created offline is recorded as an **intent**, never as a confirmed payment
    ([`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) §13.3).

### 3.4 Audit and immutability

17. Financial records are append-only. There is no delete endpoint, no soft-delete that hides money, and
    no administrative UI that removes a transaction.
18. A correction creates a reversal or adjustment entry that references the original, with an actor, a
    timestamp, a permission, and a reason.
19. The financial audit trail is stored separately from application logs and is not subject to log
    rotation ([`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) §20.2).
20. Every financial record carries `tenant_id` and is subject to hard gate 1
    ([`TENANT_ISOLATION_POLICY.md`](TENANT_ISOLATION_POLICY.md)).

### 3.5 Price history

21. An order line stores the resolved price at creation time — unit price, quantity, discounts, and the
    computed amount — not a reference that would be re-resolved later.
22. Editing a price list creates a new version; existing orders continue to reference the version they
    captured.
23. Reprinting an old nota reproduces the original amounts exactly.
24. A test asserts that mutating a price list does not change any historical order total.

### 3.6 Offline financial operations

25. The financial queue is durable and survives application restart and device reboot.
26. **The financial queue is never casually deleted.** Clearing it is an explicit, permissioned, audited
    action, never a side effect of a cache clear, a logout, or an update.
27. **Payment conflicts are never silently overwritten.** A conflict surfaces to a human with both values
    and requires a decision.
28. Retry uses exponential backoff and is bounded; an operation that cannot be delivered is escalated,
    not dropped.
29. **A duplicate order or duplicate payment caused by a retry is a financial integrity failure** and
    therefore a NO-GO.
30. The queue is partitioned per tenant and per user.
31. Sync state is visible to the user; a kasir must never believe a payment is recorded when it is still
    queued.

### 3.7 Cash

32. Shift closing records expected cash, counted cash, and the variance.
33. A variance beyond the tenant's configured threshold requires a reason and, per configuration, an
    approval.
34. Courier cash is tracked as a distinct balance from collection through hand-over, with proof at each
    transition ([`../MASTER_SOURCE.md`](../MASTER_SOURCE.md) §10).
35. A courier's outstanding cash is visible to the manager and the owner.
36. Cash movements are financial transactions and obey every rule in this document.

## 4. Verification

### 4.1 Mandatory test suite

From **Step 5** onward, a dedicated financial integrity suite runs on every pull request and every merge.
It must:

1. Submit the same payment twice with the same `client_reference` and assert exactly one payment exists.
2. Submit a burst of concurrent identical payments and assert exactly one payment exists.
3. Replay a gateway callback repeatedly and assert the state changes exactly once.
4. Submit a callback with an invalid signature and assert it is rejected and recorded.
5. Submit a callback with a mismatched amount and assert it is rejected.
6. Attempt to mark an order paid from a client claim and assert it is refused.
7. Attempt to delete a financial transaction through the API and assert it is refused.
8. Perform a refund without permission and assert it is refused; with permission and no reason, assert it
   is refused.
9. Mutate a price list and assert that historical order totals are unchanged.
10. Replay an offline financial queue after a simulated crash and assert no duplicates.
11. Force a payment conflict and assert it surfaces rather than overwriting.
12. Assert, by static analysis, that no floating-point type appears in a money path.
13. Assert that shift-closing variance is computed and recorded correctly.
14. Fail the build on any violation.

### 4.2 Review checklist

Every pull request touching money is reviewed for:

- [ ] All money values are integers; no float appears anywhere in the path.
- [ ] Every money-creating operation is idempotent, enforced at the database level.
- [ ] No endpoint accepts a client claim of payment.
- [ ] Gateway callbacks are signature-, amount-, and status-verified.
- [ ] No delete path exists for a financial record.
- [ ] Corrections create reversal or adjustment entries with actor, reason, and permission.
- [ ] Historical prices are captured, not re-resolved.
- [ ] Offline queue changes preserve durability and never permit silent overwrite.
- [ ] Rounding rules are explicit, documented, and identical across server and client.
- [ ] The financial integrity suite covers the new surface.

### 4.3 Evidence

Every evidence pack from Step 5 onward contains a financial integrity attestation with the unedited test
output ([`EVIDENCE_POLICY.md`](EVIDENCE_POLICY.md) §7).

## 5. Examples that trigger the gate

Each of the following is an automatic **NO-GO**:

- a duplicate payment created by a retry, a double tap, or a replayed callback;
- a floating-point type anywhere in a money calculation, storage, or transport path;
- an order marked paid without server-side verification;
- a financial transaction deletable through any UI or API;
- a historical order total changing after a price-list edit;
- an offline financial queue cleared as a side effect of another action;
- a payment conflict resolved by silent overwrite;
- a courier cash discrepancy the system cannot account for;
- a refund executed without permission or without a recorded reason;
- a rounding rule that makes line items and the total disagree.

---

## 6. Incident response

1. **Declare NO-GO immediately.**
2. **Stop further financial processing** on the affected path if it is deployed.
3. **Quantify the impact**: which tenants, which orders, how much money, over what period.
4. **Never fix by deleting records.** Corrections are reversals or adjustments, with reasons, preserving
   the trail.
5. **Notify affected tenants honestly**, with the amounts and the remedy.
6. **Fix the cause**, and extend the financial integrity suite in the same pull request so the defect
   cannot recur silently.
7. **Preserve sanitised evidence** and record the incident.
