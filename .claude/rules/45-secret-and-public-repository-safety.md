# Rule 45 — Secret and Public Repository Safety at Runtime

## Purpose

Rule 03 and Rule 23 fixed the public-repository authoring constraints as documentation, when there was
no runtime that could generate a `.env` file, a seed script, or a captured log. Step 3 is where those
artefacts first actually exist. This rule extends the same constraints to runtime-generated content,
because a real seed script is exactly the kind of place a real phone number quietly slips in.

Backed by Rule 03 and Rule 23; this repository remains **PUBLIC** (AMENDMENT-0001, DEC-0016).

## Hard rules

1. **Development seeds, fixtures, and factory-generated data contain only fictional data.** No real
   customer name, phone number, address, order, or credential is used to seed a local database or a test
   fixture — not "just for this branch," not "just locally," because deletion is not remediation on a
   public repository and a seed committed once is assumed indexed forever (Rule 23 of `CLAUDE.md`, Rule
   23 of `.claude/rules/`).

## Supporting expectations

- A fictional phone number, address, or name used anywhere in Step 3 runtime code or fixtures is
  recognisably fabricated — an obviously sequential, repeated, or placeholder pattern — not merely a
  number that happens not to be real; a plausible-looking fake reads as a genuine disclosure to an
  outside reader.
- No `.env` file, production configuration, private key, database dump, or backup is committed. Local
  development configuration uses obviously-fake, loopback-bound credentials that grant access to nothing
  beyond a throwaway local service (Rule 03, hard rules 9–10 and 25–28; the pattern already followed by
  `infrastructure/docker-compose.dev.yml`).
- A secret accidentally committed is treated as compromised the moment it is pushed: rotate first,
  remove second, disclose to the owner — removal alone is never reported as the fix (Rule 03, Rule 23).
- Structured logs, crash reports, and telemetry generated by Step 3 runtime code never contain a
  password, OTP, token, credential, or unredacted personal datum (Rule 03, hard rule 20; Rule 46).
- Object storage introduced for Step 3 (if any) is never publicly readable or listable for tenant data;
  private files are served only through signed, expiring URLs (Rule 06, hard rules 16–17).

## Step 3 note

No production credential, customer data, or real secret exists anywhere in this repository as of Step
3's current state. The only runtime-adjacent artefact present is a local, loopback-bound development
compose file carrying deliberately fake credentials, documented as such. This rule fixes the standard
every seed script, fixture, and configuration file introduced from here forward must meet.

## Violation handling

- **Real personal data found in a seed, fixture, or factory** — remove immediately, replace with a
  fictional and recognisably fictional value, and treat as a disclosure if it was already pushed (Rule
  23).
- **A secret, `.env` file, private key, or dump committed** — treat as compromised; rotate first, then
  remove, then disclose to the owner (Rule 03, Rule 23). Removal without rotation is not reported as a
  fix.
- **A production or staging configuration value found in a development artefact** — remove it and
  replace with an obviously fake, loopback-bound equivalent.
- **A password, OTP, token, or credential found in a log or telemetry payload** — stop, fix the logging
  path, and purge the affected log data where possible (Rule 03, Rule 46).
- **A publicly readable or listable object-storage bucket holding tenant data** — automatic `NO-GO`
  under Rule 03; revoke the exposure immediately and notify the owner.
