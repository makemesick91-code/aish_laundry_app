# Rule 46 — Runtime Observability

## Purpose

Once real requests hit a real backend, someone will eventually need to answer "who did what, to which
tenant, when" — for debugging, for an audit, for a dispute between a tenant and a customer. Observability
built without tenant and actor context from day one has to be rebuilt later, usually right after the
incident that revealed the gap. This rule fixes that foundation now, at the same time as everything else
it needs to observe.

## Hard rules

1. **Every audit event and log entry generated by the runtime is tenant-aware and actor-aware.** A log
   line or audit record states which tenant the action belongs to and which authenticated actor
   performed it; an audit trail with no actor or no tenant attached is not useful evidence and is treated
   as an incomplete implementation (Rule 03, hard rule 20 context; Rule 18, invariant 26 context).
2. **Tokens, OTPs, password-reset tokens, session cookies, and authorization headers are never written to
   logs, telemetry, crash reports, or queued-job payloads, at any log level, temporarily or permanently.**
   Debug-level logging is not an exception (Rule 03, hard rule 20; Rule 21, hard rule 18).

## Supporting expectations

- Structured logs use a consistent, parseable format across backend modules so an incident is
  investigable without guessing at ad hoc string formats.
- Personal data appearing in an error report, a crash trace, or an analytics event is redacted at source,
  not filtered downstream where a redaction gap is invisible until an incident (Rule 03, privacy
  posture).
- Observability data is tenant-scoped in storage and access exactly like business data: an operator
  investigating tenant A's incident does not incidentally see tenant B's log lines (Rule 02, Rule 39).
- Metrics and traces carry enough context to distinguish tenants and actors without carrying the
  underlying personal data itself — a tenant identifier and an actor identifier are sufficient; a full
  phone number or address in a trace attribute is not necessary and is therefore not included (Rule 21,
  data classification).
- The `packages/observability/` package is the single place this cross-cutting concern is implemented;
  individual modules consume it rather than each inventing their own logging convention (Rule 06, hard
  rule 6).

## Step 3 note

**No observability runtime exists yet.** `packages/observability/` presently contains only a
`README.md`. This rule fixes the requirement the first logging and audit implementation must satisfy
from its first commit, so that tenant and actor context is never an afterthought bolted on after a
support incident exposes the gap.

## Violation handling

- **A log entry, audit record, or metric with no tenant or actor attached** — the implementation is
  incomplete; add the missing context before relying on the record for anything.
- **A token, OTP, reset token, cookie, or authorization header found in a log, event, or crash report at
  any level** — treat as compromised; rotate the affected credential, then fix the logging path (Rule
  03, Rule 21).
- **Observability data readable across a tenant boundary** — treat as a tenant-isolation defect with the
  same severity as a business-data leak (Rule 02, Rule 48).
- **Unredacted personal data found in an error report, crash trace, or analytics event** — fix the
  redaction at source and treat as a privacy defect (Rule 03).
- **Multiple modules each implementing their own ad hoc logging instead of using the shared
  observability package** — treat as a modularity defect and consolidate before the Step closes (Rule
  06).
